# Invoices Feature — PRD

Owner: pm (Product Manager)
Date: 2026-07-01

---

## Overview

Enable users to create, view, list, filter, update, delete, and transition customer invoices. Invoices are commercial billing records derived from a customer contract and optionally seeded from the customer’s related tasks and expenses. Unlike Contracts and Expenses, invoices are editable only while still operationally open.

Each invoice belongs to exactly one customer and exactly one contract. The contract is mandatory and acts as the pricing anchor for the invoice. Invoice line items are initially suggested from:

- the linked contract itself
- the linked customer’s relevant tasks
- the linked customer’s active expenses

These suggestions are a **starting point only**. Before saving, the user may freely add, remove, re-order, or modify line items.

The Invoices page is accessible from the main sidebar and provides:

- an invoices grid/table
- filtering by customer
- filtering by status
- a create flow via `InvoiceModal`
- an edit flow for `draft` and `sent` invoices
- a per-row delete action for `draft` and `sent` invoices
- status transition actions for sending, paying, and cancelling invoices

Invoice lifecycle in v1:

- **Draft** — invoice is being prepared and may be fully edited or deleted
- **Sent** — invoice has been issued to the customer and may still be edited or deleted
- **Paid** — invoice is financially closed; it cannot be edited or deleted
- **Cancelled** — invoice is voided; it cannot be edited or deleted

The scope of v1 includes **create + read + list + filter + edit + delete + status transitions + prefill suggestions**. It does not include PDF generation, email delivery, partial payments, payment reminders, discounts, credit notes, attachments, or multi-currency conversion logic.

## 1) Data Model

### Invoice statuses

| Enum value | UI label | Meaning | Editable / Deletable |
|---|---|---|---|
| `draft` | Draft | Initial working state before sending | Yes |
| `sent` | Sent | Invoice has been issued to the customer | Yes |
| `paid` | Paid | Invoice is settled and financially closed | No |
| `cancelled` | Cancelled | Invoice has been voided | No |

### Invoices table

```sql
CREATE TYPE invoice_status AS ENUM ('draft', 'sent', 'paid', 'cancelled');

CREATE SEQUENCE IF NOT EXISTS invoice_number_seq START WITH 1 INCREMENT BY 1;

CREATE TABLE IF NOT EXISTS invoices (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  invoice_number text NOT NULL UNIQUE DEFAULT ('INV-' || LPAD(nextval('invoice_number_seq')::text, 4, '0')),
  customer_id    uuid NOT NULL REFERENCES customers(id),
  contract_id    uuid NOT NULL REFERENCES contracts(id),
  status         invoice_status NOT NULL DEFAULT 'draft',
  issue_date     date NOT NULL,
  due_date       date NOT NULL,
  notes          text,
  currency       char(3) NOT NULL DEFAULT 'NIS',
  subtotal       numeric(14,2) NOT NULL DEFAULT 0,
  tax_rate       numeric(5,2) NOT NULL DEFAULT 0,
  tax_amount     numeric(14,2) NOT NULL DEFAULT 0,
  total          numeric(14,2) NOT NULL DEFAULT 0,
  created_at     timestamptz NOT NULL DEFAULT now(),
  updated_at     timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT invoices_due_date_chk CHECK (due_date >= issue_date)
);

CREATE INDEX IF NOT EXISTS invoices_customer_id_idx  ON invoices(customer_id);
CREATE INDEX IF NOT EXISTS invoices_contract_id_idx  ON invoices(contract_id);
CREATE INDEX IF NOT EXISTS invoices_status_idx       ON invoices(status);
CREATE INDEX IF NOT EXISTS invoices_issue_date_idx   ON invoices(issue_date DESC);
```

### Invoice line items table

```sql
CREATE TABLE IF NOT EXISTS invoice_line_items (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  invoice_id   uuid NOT NULL REFERENCES invoices(id) ON DELETE CASCADE,
  sort_order   integer NOT NULL DEFAULT 0,
  description  text NOT NULL,
  quantity     numeric(10,2) NOT NULL DEFAULT 1 CHECK (quantity > 0),
  unit_price   numeric(14,2) NOT NULL CHECK (unit_price >= 0),
  amount       numeric(14,2) NOT NULL,
  source_type  text CHECK (source_type IN ('task','expense','contract','manual')),
  source_id    uuid,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS invoice_items_invoice_idx ON invoice_line_items(invoice_id);
```

### Field rules

| Field | Required | Notes |
|---|---|---|
| `id` | System | UUID generated by DB |
| `invoice_number` | System | Auto-generated sequential identifier such as `INV-0001`; never accepted from client input |
| `customer_id` | Yes | FK to `customers.id`; customer must exist |
| `customerName` | Derived | Returned by API only; never stored |
| `contract_id` | Yes | FK to `contracts.id`; contract must exist and belong to the same customer |
| `contractType` | Derived | Returned by API only from the linked contract |
| `contractTypeLabel` | Derived | Returned by API only; user-facing label for the contract type |
| `status` | System / Controlled mutation | Defaults to `draft`; only valid values: `draft`, `sent`, `paid`, `cancelled` |
| `issue_date` | Yes | Required date-only value |
| `due_date` | Yes | Required date-only value; must be `>= issue_date` |
| `notes` | No | Optional free text shown on the invoice; recommended max 4000 chars |
| `currency` | System default / Yes | Defaults from linked contract; allowed values in v1: `NIS`, `USD` |
| `subtotal` | System | Stored computed sum of line item amounts |
| `tax_rate` | Yes | Decimal percentage; defaults to `0`; recommended validation `0 <= tax_rate <= 100` |
| `tax_amount` | System | Stored computed value: `subtotal * tax_rate / 100` |
| `total` | System | Stored computed value: `subtotal + tax_amount` |
| `created_at` | System | Auto-generated on create |
| `updated_at` | System | Auto-generated on create/update/status transition |
| `lineItems` | Yes | Required non-empty array on create and update; persisted in `invoice_line_items` |

### Line item field rules

| Field | Required | Notes |
|---|---|---|
| `id` | System | UUID generated by DB |
| `invoice_id` | System | FK to parent invoice |
| `sort_order` | System / FE-controlled ordering | Determines display order in grid/detail/PDF-ready layouts later |
| `description` | Yes | Required free text |
| `quantity` | Yes | Positive decimal; defaults to `1` |
| `unit_price` | Yes | Non-negative decimal; may be `0` for informational or manually adjusted draft lines |
| `amount` | System | Computed as `quantity * unit_price`; never trusted from client |
| `source_type` | No | `task` \| `expense` \| `contract` \| `manual`; used for traceability only |
| `source_id` | No | Originating record ID when applicable |
| `created_at` | System | Auto-generated on insert |

### Line item source types

| Source type | Meaning | Typical origin |
|---|---|---|
| `contract` | Base commercial line derived from the selected contract | Lump sum total, prepaid amount, monthly retainer fee |
| `task` | Work line derived from a customer task | T&M task hours or informational task breakdown |
| `expense` | Re-billable or pass-through line derived from a customer expense | Vendor expense billed to customer |
| `manual` | User-entered ad hoc line | Discount replacement line, custom service line, adjustment |

### Prefill strategy by contract type

Prefill suggestions must be deterministic and must never write to the database.

| Contract type | Suggested contract-derived lines | Suggested task-derived lines | Suggested expense-derived lines |
|---|---|---|---|
| `lump_sum` | 1 line for `total_amount` | Include customer tasks as optional informational starting rows with `quantity = 1` and `unitPrice = 0` | Include active customer expenses matching invoice currency |
| `time_and_materials` | No separate contract base line required | One line per eligible customer task using `estimated_hours * hourly_rate` | Include active customer expenses matching invoice currency |
| `prepaid_hours` | 1 line for `amount_paid` | Include customer tasks as optional informational starting rows with `quantity = 1` and `unitPrice = 0` | Include active customer expenses matching invoice currency |
| `monthly_retainer` | 1 line for `monthly_fee` | Include customer tasks as optional informational starting rows with `quantity = 1` and `unitPrice = 0` | Include active customer expenses matching invoice currency |

**Implementation notes:**

- For T&M contracts, only tasks with `estimated_hours > 0` should become priced task suggestions.
- For non-T&M contracts, suggested task rows are informational starting points and are intentionally user-editable; users may price them, remove them, or keep them as zero-value supporting detail.
- Expenses with a different currency than the invoice currency should be excluded from automatic suggestions in v1 to avoid mixed-currency invoice totals.
- Tasks with `status = cancelled` should never be suggested. Eligible task statuses are `pending`, `in_progress`, and `done`.
- Expense suggestions should only include `expenses.status = active`.
- The DB stores snake_case columns internally; API request/response payloads should remain camelCase.
- The `gen_random_uuid()` default assumes the required UUID generation support is enabled in the database environment.

## 2) API Endpoints (NestJS)

### Base path: `/invoices`

| Method | Path | Description |
|---|---|---|
| GET | `/invoices` | List invoices (filterable) |
| GET | `/invoices/:id` | Get a single invoice with line items |
| POST | `/invoices` | Create an invoice with line items |
| PUT | `/invoices/:id` | Update an invoice and replace all line items |
| DELETE | `/invoices/:id` | Delete an invoice |
| PATCH | `/invoices/:id/status` | Transition invoice status |
| GET | `/invoices/prefill/:contractId` | Return suggested line items based on contract + customer tasks + customer expenses |

| Param | Type | Description |

Behavior:

- allowed only when current invoice status is `draft` or `sent`

- recomputes all monetary totals server-side
- preserves immutable system fields such as `id`, `invoiceNumber`, and `createdAt`

### DELETE /invoices/:id

Behavior:

- allowed only when current invoice status is `draft` or `sent`
- deletes the invoice row and all related line items via cascading delete
- should return HTTP `204 No Content` on successful deletion

Failure modes:

- HTTP `403 Forbidden` if current status is `paid` or `cancelled`
- HTTP `404 Not Found` if invoice does not exist

### Explicit non-endpoints

The following endpoints must **not** exist in v1:

- `PATCH /invoices/:id` (general partial update)
- `PATCH /invoices/:id/deactivate`
- `POST /invoices/prefill/:contractId`

## 3) Business Rules

### Mutability gate

- Invoices are editable and deletable only while `status IN ('draft', 'sent')`.
- `PUT /invoices/:id` and `DELETE /invoices/:id` must return HTTP `403 Forbidden` when `status IN ('paid', 'cancelled')`.
- The UI must hide Edit and Delete affordances for `paid` and `cancelled` invoices.
- `invoiceNumber`, `createdAt`, and historical status progression are system-controlled and must never be directly editable.

### Status transitions

- New invoices always start as `draft`.
- Only the dedicated status endpoint may change `status`.
- Valid transitions are:
  - `draft` → `sent`
  - `sent` → `paid`
  - `draft` → `cancelled`
  - `sent` → `cancelled`
  - `paid` → `cancelled`
- Any other transition must return HTTP `400 Bad Request`.
- `cancelled` is terminal in v1; once cancelled, the invoice cannot return to another state.
- There is no “partially paid”, “overdue”, “refunded”, or “archived” state in v1.

### Totals and recalculation

- Line item `amount` is always computed as `quantity * unit_price`.
- Invoice `subtotal` is always the sum of all persisted line item amounts.
- Invoice `tax_amount` is always computed as `subtotal * tax_rate / 100`.
- Invoice `total` is always computed as `subtotal + tax_amount`.
- The backend must recompute these values on every create and update and must not trust client-supplied totals.
- The API should serialize monetary values as strings to preserve decimal precision.

### Contract and customer relationship

- Every invoice must belong to exactly one customer and exactly one contract.
- The selected contract must belong to the same customer as `customerId`.
- Invoices cannot be created without a contract.
- Invoices cannot be created for deleted or missing customers/contracts.
- Because invoices are historical financial records, FK integrity should be preserved; product decisions about deleting customers/contracts with invoices should default to protecting invoice history rather than orphaning it.

### Prefill semantics

- Prefill is advisory and non-destructive.
- Calling the prefill endpoint must never create draft invoices, draft line items, audit rows, or temporary DB records.
- Suggested lines are a starting point only; the user may add, remove, modify, and reorder them before save.
- A user may create an invoice without using every suggestion.
- A user may delete all prefilled lines and replace them with manual lines, as long as at least one final line item remains before save.
- For T&M contracts, task-based suggestions should use the contract `hourly_rate`.
- For non-T&M contracts, the contract-derived commercial amount should come from the contract, while task suggestions remain informational unless the user changes them.

### Currency semantics

- Invoice `currency` defaults from the selected contract.
- Supported currencies in v1 are `NIS` and `USD`.
- Users may override invoice currency before saving, but the system does not perform FX conversion.
- Automatic expense suggestions should only include expenses whose currency already matches the invoice currency.
- If the user overrides currency after prefill, FE should either re-run prefill or warn that suggested expense lines may no longer be compatible.

### Invoice numbering

- `invoiceNumber` is auto-generated by the database sequence and is never user-supplied.
- Invoice numbers must be unique.
- The display format is `INV-0001`, `INV-0002`, etc.
- Deleted invoice numbers are not reused.

### Line item replacement and ordering

- `PUT /invoices/:id` replaces the entire line item collection.
- Existing line items should be deleted and the submitted set re-inserted inside the same transaction.
- `sortOrder` is the source of truth for row ordering.
- The FE should preserve the user’s visible line order and submit it explicitly.
- Source metadata (`sourceType`, `sourceId`) is helpful for traceability, but invoice correctness must not depend on the source records continuing to exist unchanged later.

### Date semantics and audit

- `issueDate` is the date the invoice is issued to the customer.
- `dueDate` is the payment due date and must not precede `issueDate`.
- `createdAt` is the audit timestamp for record creation.
- `updatedAt` must change on header edits, line-item replacement, and status transitions.
- Invoice list ordering should prioritize `issueDate` for business relevance.

## 4) FE Behavior

### 4a. Sidebar entry

- Add a new sidebar item: **Invoices**
- Use the same visual style and navigation behavior as the existing Customers, Contracts, and Expenses items
- Route recommendation: `/invoices`

### 4b. Invoices main view

The Invoices page is a grid/table-first screen.

Default state:

- load `GET /invoices` with no query params
- show invoices across all statuses by default
- default sort is newest issue date first (`issueDate DESC`)

Grid columns:

| Column | Behavior |
|---|---|
| Invoice # | Show `invoiceNumber`; clickable to open detail view |
| Customer | Show `customerName` |
| Contract | Show human-readable contract label or a compact `{customerName} - {contractTypeLabel}` summary |
| Status | Badge: Draft / Sent / Paid / Cancelled |
| Issue Date | Localized date based on `issueDate` |
| Due Date | Localized date based on `dueDate` |
| Total | Display formatted as `{total} {currency}` |
| Actions | View always; Edit/Delete only for `draft` and `sent` |

Total display examples:

- `6142.50 USD`
- `2500.00 NIS`

### 4c. Filters row

Controls:

- **Customer filter** — dropdown/searchable select populated from customers
- **Status filter** — dropdown with `All`, `Draft`, `Sent`, `Paid`, `Cancelled`

Behavior:

- Default status filter is `All`
- `All` → calls `GET /invoices?status=all` (or omits param and relies on default)
- If a customer is selected, include `customerId=<uuid>`
- If a status is selected, include `status=<value>`
- Filters may be combined with `contractId` later if the screen gains a contract-level filter control
- Clearing a filter removes only that filter and refreshes the list

### 4d. Create/Edit button and modal

- Primary button in the top-right, consistent with the existing screen patterns
- Button label: **Create Invoice**
- Opens `InvoiceModal`
- The same modal is reused for edit mode when invoice status is `draft` or `sent`
- Recommended flow is 3-step for clarity

#### Step 1 — Invoice basics

Fields:

| Field | Control | Notes |
|---|---|---|
| Customer | Searchable select | Required |
| Contract | Searchable select | Required; options filtered by selected customer |
| Issue Date | Date picker | Required |
| Due Date | Date picker | Required; must be on/after issue date |
| Currency | Select | Required; default from contract |
| Tax Rate | Decimal number input | Required; default `0` |
| Notes | Textarea | Optional |

Behavior:

- Contract selector is disabled until a customer is selected
- Changing the customer resets the selected contract
- Once a contract is chosen, FE calls `GET /invoices/prefill/:contractId`
- Continue button is disabled until required fields are valid

#### Step 2 — Line items

The line items step is initialized from prefill suggestions and remains fully editable.

Table columns:

| Column | Behavior |
|---|---|
| Description | Editable text input |
| Quantity | Editable decimal input |
| Unit Price | Editable decimal input |
| Amount | Read-only computed row total |
| Source | Optional small label/icon for `task` / `expense` / `contract` / `manual` |
| Actions | Remove row |

Required interactions:

- User can add a blank manual row
- User can edit any prefilled row, including description, quantity, and unit price
- User can remove any row, including prefilled rows
- User can reorder rows; FE should persist the resulting `sortOrder`
- Row amount updates live as `quantity * unitPrice`
- Step should block progression if no line items remain

#### Step 3 — Review and totals

Display a read-only summary before save:

- invoice header summary (customer, contract, issue date, due date, currency, tax rate)
- line items in final order
- subtotal
- tax amount
- total

Submit flow:

1. User clicks **Create Invoice** or **Save Changes**
2. FE sends `POST /invoices` or `PUT /invoices/:id`
3. On success: close modal, show success toast, refresh list
4. On validation error: keep modal open and show inline errors
5. On `403`: show immutable-state error and refresh the invoice state from server
6. On `409`: keep modal open and explain the contract/customer mismatch

### 4e. Row actions and status transitions

Per-row actions:

- **View** — always available
- **Edit** — available only when `status = draft` or `sent`
- **Delete** — available only when `status = draft` or `sent`

Delete flow:

- opens a confirmation dialog
- confirmation copy should make mutability rules explicit, e.g.:
  - **Title:** Delete invoice?
  - **Body:** `This will permanently delete invoice INV-0007. Paid and cancelled invoices cannot be deleted.`
- confirm action calls `DELETE /invoices/:id`
- on success: remove row from grid and show success toast

Status transition actions:

- **Mark as Sent** — available when `status = draft`
- **Mark as Paid** — available when `status = sent`
- **Cancel Invoice** — available when `status = draft`, `sent`, or `paid`

UX note:

- Because `paid` invoices are not editable, the paid-state cancel action should appear in the read-only detail view rather than relying on the edit modal.

### 4f. Read-only detail view

Because the API includes `GET /invoices/:id`, FE should provide a detail surface (drawer, modal, or dedicated page).

Recommended behavior:

- clicking the invoice number or row opens a detail drawer/page
- the detail surface displays:
  - invoice number
  - customer
  - contract type label
  - status
  - issue date
  - due date
  - notes
  - currency
  - subtotal
  - tax rate
  - tax amount
  - total
  - full ordered line items list
  - createdAt
  - updatedAt
- Edit button appears only for `draft` and `sent`
- Delete button appears only for `draft` and `sent`
- Cancel action appears for `draft`, `sent`, and `paid`

### 4g. Badge variants

Invoice statuses require expanding the shared `Badge` component.

Current variants already in use:

- `default` — slate
- `primary` — indigo

New variants required for invoices:

- `success` — green: `bg-emerald-100 text-emerald-700`
- `danger` — rose: `bg-rose-100 text-rose-700`

Status-to-badge mapping:

| Status | Badge variant |
|---|---|
| Draft | `default` |
| Sent | `primary` |
| Paid | `success` |
| Cancelled | `danger` |

### 4h. Empty, loading, and error states

- Empty default state: show “No invoices yet” with Create Invoice CTA
- Empty filtered state: show “No invoices match the current filters”
- Loading state: skeleton rows/spinner consistent with existing tables
- API errors: show inline toast/banner; preserve filters and unsaved modal state when possible
- If prefill fails, keep the user on the modal and show a recoverable error with retry action

## 5) Acceptance Criteria

### Data / API

- [ ] An invoice can be created for any existing contract belonging to an existing customer.
- [ ] New invoices are created with `status = draft`, a system-generated `invoiceNumber`, and system-generated audit timestamps.
- [ ] `invoiceNumber` is unique and formatted sequentially as `INV-0001`, `INV-0002`, etc.
- [ ] `GET /invoices/prefill/:contractId` returns suggested line items based on the linked contract, eligible customer tasks, and eligible customer expenses.
- [ ] Prefill does not write anything to the database.
- [ ] The API exposes: `GET /invoices`, `GET /invoices/:id`, `GET /invoices/prefill/:contractId`, `POST /invoices`, `PUT /invoices/:id`, `DELETE /invoices/:id`, and `PATCH /invoices/:id/status`.
- [ ] `PUT /invoices/:id` replaces all line items transactionally.
- [ ] Totals (`subtotal`, `taxAmount`, `total`) are always recomputed server-side.
- [ ] `GET /invoices` can be filtered by `customerId`, `contractId`, and `status`.
- [ ] Invalid status transitions return HTTP `400 Bad Request`.
- [ ] Editing or deleting a `paid` or `cancelled` invoice returns HTTP `403 Forbidden`.

### Business behavior

- [ ] Line items are auto-populated as a starting point and can be manually edited before saving.
- [ ] Users can add, remove, modify, and reorder line items prior to save.
- [ ] `draft` invoices can be edited or deleted.
- [ ] `sent` invoices can be edited or deleted.
- [ ] `paid` invoices cannot be edited or deleted.
- [ ] `cancelled` invoices cannot be edited or deleted.
- [ ] Status transitions follow the allowed flow: `draft → sent`, `sent → paid`, and `draft|sent|paid → cancelled`.
- [ ] Contract/customer mismatch is rejected by the API.
- [ ] Due date cannot be earlier than issue date.
- [ ] Mixed-currency automatic expense suggestion is not allowed in v1.

### Frontend

- [ ] The sidebar contains an Invoices navigation item.
- [ ] The Invoices page shows Invoice #, Customer, Contract, Status, Issue Date, Due Date, Total, and Actions.
- [ ] The grid can be filtered by customer and status.
- [ ] The Create Invoice button opens a multi-step `InvoiceModal`.
- [ ] Step 1 captures customer, contract, issue date, due date, currency, tax rate, and notes.
- [ ] Step 2 displays editable prefilled line items and supports add/remove/reorder actions.
- [ ] Step 3 shows a review summary with subtotal, tax, and total.
- [ ] Edit and Delete actions are shown only for `draft` and `sent` invoices.
- [ ] Status badges use Draft=`default`, Sent=`primary`, Paid=`success`, Cancelled=`danger`.
- [ ] The shared `Badge` component is extended with `success` and `danger` variants.
- [ ] Paid invoices can still be cancelled from a read-only detail surface.
- [ ] On successful create, update, delete, or status transition, the grid refreshes and reflects the latest invoice state.
