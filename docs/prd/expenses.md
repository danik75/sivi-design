Owner: pm (Product Manager)

## Overview

Enable users to create, view, list, filter, and deactivate 3rd party expenses. Expenses represent external costs paid to vendors or suppliers and may optionally be associated with a customer. Expense records are immutable after creation: they cannot be edited or deleted, only deactivated.

The feature introduces a new **Expenses** item in the main sidebar and provides:

- an expenses grid/table
- filtering by customer
- filtering by category
- an `Active / All` status toggle (default: `Active`)
- a create flow via `ExpenseModal`
- a per-row deactivate action

The scope of v1 is intentionally constrained to **create + read + list + deactivate**. There is no edit flow, no delete flow, no bulk operations, and no status transitions other than deactivation.

Expenses are operational records rather than commercial agreements:

- they always belong to a vendor
- they always carry a positive monetary amount
- they always have an incurred date
- they may or may not be linked to a customer

When an expense is linked to a customer, the API response should include `customerName` for UI display. When no customer is linked, `customerId` and `customerName` are both `null`.

## 1) Data Model

### Expense categories

| Enum value | UI label | Meaning |
|---|---|---|
| `software` | Software | SaaS subscriptions, hosting, tools, licenses |
| `hardware` | Hardware | Devices, peripherals, equipment |
| `subcontractor` | Subcontractor | External freelancer or agency costs |
| `travel` | Travel | Transport, lodging, travel-related reimbursements |
| `office` | Office | Office supplies, workspace-related costs |
| `other` | Other | Any expense that does not fit the predefined categories |

### Expenses table

```sql
CREATE TYPE expense_category AS ENUM (
  'software',
  'hardware',
  'subcontractor',
  'travel',
  'office',
  'other'
);

CREATE TYPE expense_status AS ENUM ('active', 'inactive');

CREATE TABLE IF NOT EXISTS expenses (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  vendor      text NOT NULL,
  amount      numeric(14,2) NOT NULL CHECK (amount > 0),
  currency    char(3) NOT NULL DEFAULT 'NIS',
  date        date NOT NULL,
  category    expense_category NOT NULL,
  description text,
  customer_id uuid REFERENCES customers(id) ON DELETE SET NULL,
  status      expense_status NOT NULL DEFAULT 'active',
  created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS expenses_customer_id_idx ON expenses(customer_id);
CREATE INDEX IF NOT EXISTS expenses_status_idx      ON expenses(status);
CREATE INDEX IF NOT EXISTS expenses_date_idx        ON expenses(date DESC);
CREATE INDEX IF NOT EXISTS expenses_category_idx    ON expenses(category);
```

### Field rules

| Field | Required | Notes |
|---|---|---|
| `id` | System | UUID generated by DB |
| `vendor` | Yes | Required free text; vendor/supplier name; recommended max 255 chars |
| `amount` | Yes | Positive decimal; stored as `numeric(14,2)`; max 2 fraction digits |
| `currency` | System default / Yes | Defaults to `NIS`; allowed values in v1: `NIS`, `USD` |
| `date` | Yes | Date the expense was incurred; date-only value |
| `category` | Yes | `software` \| `hardware` \| `subcontractor` \| `travel` \| `office` \| `other` |
| `description` | No | Optional notes/free text; recommended max 2000 chars |
| `customer_id` | No | Optional FK to `customers.id`; may be null |
| `status` | System | Defaults to `active`; only valid values: `active`, `inactive` |
| `created_at` | System | Auto-generated on create; immutable |
| `customerName` | Derived | Returned by API only when customer is linked; never stored in `expenses` table |
| `categoryLabel` | Derived | Returned by API only; mapped from enum to UI label; never stored |

### Category label mapping

| Enum | UI Label |
|---|---|
| `software` | Software |
| `hardware` | Hardware |
| `subcontractor` | Subcontractor |
| `travel` | Travel |
| `office` | Office |
| `other` | Other |

**Implementation note:** the schema stores raw enum values, while the API must also provide display-friendly fields such as `categoryLabel`. The `gen_random_uuid()` default assumes the database environment has the required UUID generation support enabled.

## 2) API Endpoints (NestJS)

### Base path: `/expenses`

| Method | Path | Description |
|---|---|---|
| GET | `/expenses` | List expenses (filterable) |
| GET | `/expenses/:id` | Get a single expense |
| POST | `/expenses` | Create an expense |
| PATCH | `/expenses/:id/deactivate` | Deactivate an expense (only allowed mutation) |

### Query params for GET /expenses

| Param | Type | Description |
|---|---|---|
| `customer_id` | uuid | Filter by customer |
| `status` | `active` \| `inactive` \| `all` | Default: `active` |
| `category` | string | Filter by category |

### POST /expenses

`status` and `createdAt` are system-managed and must not be accepted from the client.

Create request DTOs should be camelCase for consistency with response DTOs.

#### Create expense linked to a customer

```json
{
  "vendor": "AWS",
  "amount": "250.00",
  "currency": "USD",
  "date": "2026-06-15",
  "category": "software",
  "description": "Monthly cloud hosting",
  "customerId": "4a4db57c-fd55-482a-9a60-7a7db6efb482"
}
```

#### Create expense without a customer

```json
{
  "vendor": "WeWork",
  "amount": "420.00",
  "currency": "NIS",
  "date": "2026-06-20",
  "category": "office",
  "description": "Meeting room booking",
  "customerId": null
}
```

Successful create response:

- HTTP `201 Created`
- returns the created expense DTO including `categoryLabel`

Validation failures:

- HTTP `400 Bad Request` for missing/invalid fields
- HTTP `404 Not Found` if `customerId` is provided but no customer exists for that ID

Example `400 Bad Request` response:

```json
{
  "statusCode": 400,
  "message": [
    "amount must be greater than 0",
    "currency must be one of the following values: NIS, USD"
  ],
  "error": "Bad Request"
}
```

### PATCH /expenses/:id/deactivate

Behavior:

- sets `status` from `active` to `inactive`
- does not modify any other field
- should be implemented as an idempotent endpoint; calling it for an already inactive expense returns `200` with unchanged inactive resource

Example request:

```http
PATCH /expenses/2a8e9916-fd7d-4ba7-8c12-55dc56e0d29f/deactivate
```

Example response:

```json
{
  "id": "2a8e9916-fd7d-4ba7-8c12-55dc56e0d29f",
  "vendor": "AWS",
  "amount": "250.00",
  "currency": "USD",

  "createdAt": "2026-07-01T08:10:22.000Z"
}
```

### Explicit non-endpoints

The following endpoints must **not** exist in v1:

- `PUT /expenses/:id`
- `PATCH /expenses/:id` (general update)
- `DELETE /expenses/:id`

## 3) Business Rules

### Immutability

- Expenses are append-only records after creation.
- The only permitted post-create mutation is deactivation.
- Vendor, amount, currency, incurred date, category, description, customer association, and creation timestamp are immutable after create.
- The UI must never expose edit or delete affordances.
- The backend must never register general update or delete routes for expenses.

### Customer relationship

- Customer association is optional.
- An expense may be created with `customerId = null`.
- If `customerId` is provided, it must reference an existing customer.
- Because the FK uses `ON DELETE SET NULL`, deleting a linked customer must not delete historical expense records.
- If a previously linked customer is removed, the expense remains valid and should subsequently return `customerId = null` and `customerName = null`.

### Monetary validation

- `amount` must always be greater than `0`.
- Amounts are stored as `numeric(14,2)` and should be validated as positive decimals with up to 2 fractional digits.
- Zero, negative, and non-numeric amounts must be rejected.
- The API should return amounts as strings in DTOs to preserve exact decimal formatting.

### Currency semantics

- Supported currencies in v1 are limited to `NIS` and `USD`.
- `currency` defaults to `NIS` when omitted during create.
- Currency values must always be uppercase 3-character codes in API payloads and responses.
- The FE should format amounts as `{amount} {currency}` in list and detail views unless broader localization rules are introduced later.

### Date semantics

- `date` is the date the expense was incurred, not the date the record was entered into the system.
- `date` is required for every expense.
- FE should collect it as a date-only value and BE should store it in the SQL `date` column.
- `createdAt` remains the audit timestamp for record creation and must not be used as a substitute for the incurred date.

### Category semantics

- Every expense must belong to exactly one category.
- The stored enum value is the source of truth.
- The UI must show the human-readable label, not the raw enum value, in grids, detail views, confirmation dialogs, and toasts.
- `other` exists as a catch-all category and must remain available in both API validation and FE selection controls.

### Status semantics

- New expenses always start as `active`.
- Status values are limited to `active` and `inactive`.
- There is no ÔÇťdraftÔÇŁ, ÔÇťpending approvalÔÇŁ, ÔÇťreimbursedÔÇŁ, or ÔÇťarchivedÔÇŁ state in v1.
- The default list experience shows only active expenses.
- Deactivation is a manual business action and is the only supported lifecycle transition.

### Audit and chronology

- `createdAt` is system-generated at creation time and is the source of truth for when the expense record entered the system.
- Users cannot override `createdAt` in v1.
- Expense list ordering should prioritize the incurred `date` for business relevance, while `createdAt` remains available for audit/detail views.

## 4) FE Behavior

### 4a. Sidebar entry

- Add a new sidebar item: **Expenses**
- Use the same visual style and navigation behavior as the existing Customers and Contracts items
- Route recommendation: `/expenses`

### 4b. Expenses main view

The Expenses page is a grid/table-first screen.

Default state:

- load `GET /expenses` with no query params
- show only active expenses by default
- default sort is newest incurred date first (`date DESC`)

Grid columns:

| Column | Behavior |
|---|---|
| Vendor | Plain text vendor/supplier name |
| Category | Human-readable label from `categoryLabel` |
| Amount | Display formatted as `{amount} {currency}` |
| Customer | Show linked `customerName`; em dash when no customer linked |
| Date | Localized date based on incurred `date` |
| Status | Badge: Active / Inactive |
| Actions | Deactivate action only for active rows |

Amount display examples:

- `250.00 USD`
- `420.00 NIS`

### 4c. Filters row

Controls:

- **Customer filter** ÔÇö dropdown/searchable select populated from customers
- **Category filter** ÔÇö dropdown populated from the 6 supported categories
- **Status toggle** ÔÇö `Active / All`

Behavior:

- Default is `Active`
- `Active` Ôćĺ calls `GET /expenses?status=active` (or omits param and relies on default)
- `All` Ôćĺ calls `GET /expenses?status=all`
- If a customer is selected, include `customer_id=<uuid>` in the request
- If a category is selected, include `category=<enum>` in the request
- Filters may be combined, e.g. customer + category + status
- Clearing a filter removes only that filter and refreshes the list

Note: the backend supports `status=inactive`, but the v1 UI only needs the `Active / All` toggle.

### 4d. Create button and modal

- Primary button in the top-right, consistent with the existing Customers/Contracts pattern
- Button label: **Create Expense**
- Opens `ExpenseModal`

Modal fields:

| Field | Control | Notes |
|---|---|---|
| Vendor | Text input | Required |
| Category | Select | Required; uses label map |
| Amount | Decimal number input | Required; positive only |
| Currency | Select | Required; options: `NIS`, `USD`; default `NIS` |
| Date | Date picker | Required; incurred date |
| Customer | Searchable select | Optional |
| Description | Textarea | Optional |

Validation:

- Vendor is required and cannot be blank/whitespace-only
- Category is required
- Amount is required and must be `> 0`
- Currency must be either `NIS` or `USD`
- Date is required
- Customer is optional

Submit flow:

1. User clicks **Create Expense**
2. FE sends `POST /expenses`
3. On success: close modal, show success toast, refresh list
4. On validation error: keep modal open and show field-level and/or form-level errors
5. On customer lookup failure (`404`): keep modal open and show an inline error for the customer field or a form error

### 4e. Deactivate action

- Available as a per-row button/icon only when `status = active`
- Opens confirmation dialog before calling the API
- Confirmation copy should make immutability explicit, e.g.:
  - **Title:** Deactivate expense?
  - **Body:** `This will mark the AWS expense from 2026-06-15 as inactive. Expenses cannot be edited or deleted after creation.`
- Confirm action calls `PATCH /expenses/:id/deactivate`
- On success: refresh row/list and show success toast
- Inactive rows should not show an active Deactivate button

### 4f. Read-only detail view

Because the API includes `GET /expenses/:id`, FE should provide a read-only detail surface (drawer, modal, or dedicated page).

Recommended behavior:

- clicking the vendor name or row opens a read-only detail drawer
- the detail surface displays:
  - vendor
  - category label
  - amount + currency
  - incurred date
  - customer (if linked)
  - description
  - status
  - createdAt
- no edit button and no delete button
- if expense is active, a Deactivate action may be repeated from the detail surface

### 4g. Empty, loading, and error states

- Empty active state: show ÔÇťNo active expenses yetÔÇŁ with Create Expense CTA
- Empty filtered state: show ÔÇťNo expenses match the current filtersÔÇŁ
- Loading state: skeleton rows/spinner consistent with existing tables
- API errors: show inline toast/banner; preserve filters and modal form state when possible

## 5) Acceptance Criteria

### Data / API

- [ ] An expense can be created with all required fields and an optional customer association.
- [ ] An expense can be created without a customer association.
- [ ] New expenses are created with `status = active` and a system-generated `createdAt`.
- [ ] The API exposes only: `GET /expenses`, `GET /expenses/:id`, `POST /expenses`, and `PATCH /expenses/:id/deactivate`.
- [ ] The API does **not** expose `PUT /expenses/:id`, generic `PATCH /expenses/:id`, or `DELETE /expenses/:id`.
- [ ] `GET /expenses` defaults to showing only active expenses.
- [ ] `GET /expenses?status=all` returns both active and inactive expenses.
- [ ] `GET /expenses` can be filtered by `customer_id` and/or `category`.
- [ ] Expense response DTOs use camelCase and include `categoryLabel`.
- [ ] When an expense is linked to a customer, the API response includes `customerName`; otherwise `customerId` and `customerName` are `null`.
- [ ] `amount` is validated as a positive decimal and returned as a string in the response.
- [ ] `currency` is limited to `NIS` and `USD`.

### Business behavior

- [ ] Once created, an expense cannot be edited through either UI or API.
- [ ] Once created, an expense cannot be deleted through either UI or API.
- [ ] An expense can be deactivated, and its status changes to `inactive`.
- [ ] Deactivation does not modify any field other than `status`.
- [ ] Customer association remains optional throughout the lifecycle of the expense.
- [ ] `date` is treated as the incurred date and is distinct from `createdAt`.

### Frontend

- [ ] The sidebar contains an Expenses navigation item.
- [ ] The Expenses page shows active expenses by default.
- [ ] Switching the status toggle to **All** shows inactive expenses as well.
- [ ] The grid can be filtered by customer and category, independently or together.
- [ ] The grid shows Vendor, Category, Amount, Customer, Date, Status, and Actions.
- [ ] Amount is displayed with currency (for example, `250.00 NIS`).
- [ ] Category is shown using the label map (for example, `Software`), not the raw enum value.
- [ ] Unlinked expenses show an em dash in the Customer column.
- [ ] The Create Expense button opens `ExpenseModal`.
- [ ] The modal contains Vendor, Category, Amount, Currency, Date, Customer, and Description fields.
- [ ] Currency defaults to `NIS` in the create flow.
- [ ] Active rows expose a Deactivate action.
- [ ] No Edit or Delete actions are shown anywhere on the Expenses screen.
- [ ] On successful create or deactivate, the grid refreshes and reflects the latest expense state.
