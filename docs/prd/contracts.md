# Contracts Feature — PRD

Owner: pm (Product Manager)
Date: 2026-07-01

---

## Overview

Enable users to create, view, list, filter, and deactivate customer contracts. Contracts define the commercial agreement between the company and a customer and are immutable after creation: they cannot be edited or deleted, only deactivated.

The feature supports four professional contract models:

- **Lump Sum** — a fixed total project price
- **Time & Materials (T&M)** — billing by actual hours worked at an agreed hourly rate
- **Prepaid Hours Block** — a pre-purchased fixed number of hours for a total paid amount
- **Monthly Retainer** — a recurring monthly fee that includes a fixed number of hours per month

Each contract belongs to exactly one customer. The contract display name is derived as:

`{customerName} - {contractTypeLabel}`

This name is **computed by the API** and is **not stored in the database**.

The Contracts page is accessible from the main sidebar and provides:

- a contracts grid/table
- filtering by customer
- an `Active / All` status toggle (default: `Active`)
- a create flow via `ContractModal`
- a per-row deactivate action

The scope of v1 is intentionally constrained to **create + read + list + deactivate**. There is no edit flow, no delete flow, and no bulk operations.

## 1) Data Model

### Contract types

| Enum value | UI label | Commercial meaning | Required extra fields |
|---|---|---|---|
| `lump_sum` | Lump Sum | One fixed total amount for the full project scope | `total_amount`, `currency` |
| `time_and_materials` | Time & Materials (T&M) | Billing by actual worked hours at an agreed rate | `hourly_rate`, `currency` |
| `prepaid_hours` | Prepaid Hours Block | Customer prepays a fixed amount for a fixed number of hours | `hours_purchased`, `amount_paid`, `currency` |
| `monthly_retainer` | Monthly Retainer | Monthly recurring fee covering a fixed number of hours per month | `monthly_fee`, `hours_per_month`, `currency` |

### Contracts table

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TYPE contract_type AS ENUM (
  'lump_sum',
  'time_and_materials',
  'prepaid_hours',
  'monthly_retainer'
);

CREATE TYPE contract_status AS ENUM ('active', 'inactive');

CREATE TABLE IF NOT EXISTS contracts (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id     uuid NOT NULL REFERENCES customers(id),
  type            contract_type NOT NULL,
  status          contract_status NOT NULL DEFAULT 'active',
  description     text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  expires_at      timestamptz,                        -- optional expiration date

  -- Lump Sum
  total_amount    numeric(14,2),

  -- Time & Materials
  hourly_rate     numeric(10,2),

  -- Prepaid Hours Block
  hours_purchased numeric(10,2),
  amount_paid     numeric(14,2),

  -- Monthly Retainer
  monthly_fee     numeric(14,2),
  hours_per_month numeric(10,2),

  -- Shared currency for all monetary contract types
  currency        char(3),                           -- ISO 4217, e.g. 'USD'

  CONSTRAINT contracts_total_amount_positive_chk
    CHECK (total_amount IS NULL OR total_amount > 0),
  CONSTRAINT contracts_hourly_rate_positive_chk
    CHECK (hourly_rate IS NULL OR hourly_rate > 0),
  CONSTRAINT contracts_hours_purchased_positive_chk
    CHECK (hours_purchased IS NULL OR hours_purchased > 0),
  CONSTRAINT contracts_amount_paid_positive_chk
    CHECK (amount_paid IS NULL OR amount_paid > 0),
  CONSTRAINT contracts_monthly_fee_positive_chk
    CHECK (monthly_fee IS NULL OR monthly_fee > 0),
  CONSTRAINT contracts_hours_per_month_positive_chk
    CHECK (hours_per_month IS NULL OR hours_per_month > 0)
);

CREATE INDEX IF NOT EXISTS contracts_customer_id_idx
  ON contracts(customer_id);

CREATE INDEX IF NOT EXISTS contracts_status_idx
  ON contracts(status);

CREATE INDEX IF NOT EXISTS contracts_created_at_idx
  ON contracts(created_at DESC);

-- Only one active contract per customer/type combination
CREATE UNIQUE INDEX IF NOT EXISTS contracts_active_customer_type_uidx
  ON contracts (customer_id, type)
  WHERE status = 'active';
```

### Field rules

| Field | Required | Notes |
|---|---|---|
| `id` | System | UUID generated by DB |
| `customer_id` | Yes | FK to `customers.id`; customer must exist |
| `name` | Derived | Returned by API only; computed as `{customerName} - {contractTypeLabel}`; never stored |
| `type` | Yes | `lump_sum` \| `time_and_materials` \| `prepaid_hours` \| `monthly_retainer` |
| `status` | System | Defaults to `active`; only valid values: `active`, `inactive` |
| `description` | Yes | Free-text contract summary; shown in grid/detail; recommended max 2000 chars |
| `created_at` | System | Auto-generated on create; immutable |
| `expires_at` | No | Optional expiration date; FE treats as date-only; API stores as ISO timestamp |
| `currency` | Conditional | Required for all monetary contract types; must be uppercase ISO 4217 code (3 chars) |
| `total_amount` | Conditional | Required only for `lump_sum`; positive decimal, 2 fraction digits |
| `hourly_rate` | Conditional | Required only for `time_and_materials`; positive decimal, 2 fraction digits |
| `hours_purchased` | Conditional | Required only for `prepaid_hours`; positive decimal; supports fractional hours |
| `amount_paid` | Conditional | Required only for `prepaid_hours`; positive decimal, 2 fraction digits |
| `monthly_fee` | Conditional | Required only for `monthly_retainer`; positive decimal, 2 fraction digits |
| `hours_per_month` | Conditional | Required only for `monthly_retainer`; positive decimal; supports fractional hours |

### Type-specific field matrix

| Type | Must be present | Must be null / omitted |
|---|---|---|
| `lump_sum` | `total_amount`, `currency` | `hourly_rate`, `hours_purchased`, `amount_paid`, `monthly_fee`, `hours_per_month` |
| `time_and_materials` | `hourly_rate`, `currency` | `total_amount`, `hours_purchased`, `amount_paid`, `monthly_fee`, `hours_per_month` |
| `prepaid_hours` | `hours_purchased`, `amount_paid`, `currency` | `total_amount`, `hourly_rate`, `monthly_fee`, `hours_per_month` |
| `monthly_retainer` | `monthly_fee`, `hours_per_month`, `currency` | `total_amount`, `hourly_rate`, `hours_purchased`, `amount_paid` |

**Implementation note:** the DB stores all type-specific fields in one table using nullable columns. The API and UI must enforce the type-specific requirements above at validation time.

## 2) API Endpoints (NestJS)

### Base path: `/contracts`

| Method | Path | Description |
|---|---|---|
| GET | `/contracts` | List contracts (filterable) |
| GET | `/contracts/:id` | Get a single contract |
| POST | `/contracts` | Create a contract |
| PATCH | `/contracts/:id/deactivate` | Deactivate a contract (only allowed mutation) |

### Query params for GET /contracts

| Param | Type | Description |
|---|---|---|
| `customer_id` | uuid | Filter by customer |
| `status` | `active` \| `inactive` \| `all` | Default: `active` |

Successful create response:

- HTTP `201 Created`
- returns the created contract DTO including computed `name`

Validation failures:

- HTTP `400 Bad Request` for missing/invalid fields
- HTTP `404 Not Found` if `customer_id` does not exist
- HTTP `409 Conflict` if an **active** contract already exists for the same `(customer_id, type)` combination

Example `409 Conflict` response:

```json
{
  "statusCode": 409,
  "message": "An active Time & Materials (T&M) contract already exists for this customer.",
  "error": "Conflict"
}
```

### PATCH /contracts/:id/deactivate

Behavior:

- sets `status` from `active` to `inactive`
- does not modify any other field
- should be implemented as an idempotent endpoint; calling it for an already inactive contract returns `200` with unchanged inactive resource

### Explicit non-endpoints

The following endpoints must **not** exist in v1:

- `PUT /contracts/:id`
- `PATCH /contracts/:id` (general update)
- `DELETE /contracts/:id`

## 3) Business Rules

### Immutability

- Contracts are append-only records after creation.
- The only permitted post-create mutation is deactivation.
- Contract type, customer, description, financial terms, currency, creation date, and expiration date are immutable after create.
- The UI must never expose edit or delete affordances.
- The backend must never register general update or delete routes for contracts.

### Uniqueness

- Only one **active** contract may exist for the same `(customer_id, type)` pair.
- This must be enforced in two places:
  1. **Database** via partial unique index
  2. **API** via pre-insert validation and `409 Conflict` response
- Inactive contracts do not participate in the uniqueness restriction.
- After an active contract is deactivated, a new active contract of the same type may be created for the same customer.

### Derived naming

- Contract `name` is assembled dynamically as `{customerName} - {contractTypeLabel}`.
- `customerName` comes from the related customer record.
- `contractTypeLabel` must be the professional UI label, not the raw enum.
- The derived name is used consistently in list views, detail views, success toasts, confirmation dialogs, and API DTOs.

### Type-specific validation

- Exactly one contract type is chosen at create time.
- Required fields depend on that type.
- Fields that do not belong to the selected type must be omitted or sent as `null` and ignored/rejected consistently by BE validation.
- All monetary values must be positive decimals with up to 2 fractional digits.
- Hour quantities may be decimals to support fractional hours.
- `currency` is required for all four current contract types.

### Status semantics

- New contracts always start as `active`.
- Status values are limited to `active` and `inactive`.
- There is no “draft”, “expired”, or “archived” state in v1.
- Deactivation is a business decision, not an automatic process.

### Expiration semantics

- `expires_at` is optional.
- In v1, expiration is informational only and does **not** automatically switch the contract to `inactive`.
- An active contract with a past `expires_at` remains active until the deactivate action is executed.
- FE should display the expiration date when present; absence means open-ended duration.
- FE collects expiration as a date-only input; BE stores a normalized ISO timestamp.

### Customer relationship

- Every contract must belong to an existing customer.
- Contracts cannot be created without a customer.
- Since contracts are immutable records, customer deletion behavior must preserve referential integrity. For v1, deleting a customer that has contracts should be blocked by the FK unless product requirements change later.

### Audit and chronology

- `created_at` is system-generated at creation time and is the source of truth for when the contract record entered the system.
- Users cannot backdate or override `created_at` in v1.
- Lists default to newest created contracts first.

## 4) FE Behavior

### 4a. Sidebar entry

- Add a new sidebar item: **Contracts**
- Use the same visual style and navigation behavior as the existing Customers item
- Route recommendation: `/contracts`

### 4b. Contracts main view

The Contracts page is a grid/table-first screen.

Default state:

- load `GET /contracts` with no query params
- show only active contracts by default
- default sort is newest first (`created_at DESC`)

Grid columns:

| Column | Behavior |
|---|---|
| Name | Computed value `{customerName} - {contractTypeLabel}` |
| Type | Human-readable label (`Lump Sum`, `Time & Materials (T&M)`, etc.) |
| Description | Truncated to 1–2 lines with tooltip/title on hover |
| Commercial Terms | Summary of type-specific financial fields |
| Status | Badge: Active / Inactive |
| Created At | Localized date/time |
| Expires At | Localized date when present; em dash when absent |
| Actions | Deactivate action only for active rows |

Commercial Terms display examples:

- Lump Sum → `USD 15,000.00`
- Time & Materials → `USD 120.00 / hour`
- Prepaid Hours Block → `40h for USD 4,000.00`
- Monthly Retainer → `USD 5,000.00 / month · 25h/month`

### 4c. Filters row

Controls:

- **Customer filter** — dropdown/searchable select populated from customers
- **Status toggle** — `Active / All`

Behavior:

- Default is `Active`
- `Active` → calls `GET /contracts?status=active` (or omits param and relies on default)
- `All` → calls `GET /contracts?status=all`
- If a customer is selected, include `customer_id=<uuid>` in the request
- Clearing the customer filter resets to all customers

Note: the backend supports `status=inactive`, but the v1 UI only needs the `Active / All` toggle.

### 4d. Create button and modal

- Primary button in the top-right, consistent with the Customer screen pattern
- Button label: **Create Contract**
- Opens `ContractModal`
- Modal may be multi-step or adaptive; recommended flow is 2-step for clarity

#### Step 1 — Contract basics

Fields:

| Field | Control | Notes |
|---|---|---|
| Customer | Searchable select | Required |
| Contract Type | Select / cards | Required; locks step 2 fields |
| Description | Textarea | Required |
| Expiration Date | Date picker | Optional |

Behavior:

- Continue button disabled until required fields are valid
- Type selection immediately determines the fields shown in step 2
- Show a helper preview of the computed contract name once customer + type are selected

#### Step 2 — Commercial terms (conditional)

| Type | Fields shown |
|---|---|
| Lump Sum | `total_amount`, `currency` |
| Time & Materials (T&M) | `hourly_rate`, `currency` |
| Prepaid Hours Block | `hours_purchased`, `amount_paid`, `currency` |
| Monthly Retainer | `monthly_fee`, `hours_per_month`, `currency` |

Validation:

- required fields must be present before submit
- monetary inputs use decimal number fields
- hour inputs allow decimal values
- currency should be a 3-letter uppercase select/input (prefer select if supported)

Submit flow:

1. User clicks **Create**
2. FE sends `POST /contracts`
3. On success: close modal, show success toast, refresh list
4. On `409`: keep modal open and show inline/form-level error explaining that an active contract of this type already exists for the selected customer

### 4e. Deactivate action

- Available as a per-row button/icon only when `status = active`
- Opens confirmation dialog before calling the API
- Confirmation copy should make immutability explicit, e.g.:
  - **Title:** Deactivate contract?
  - **Body:** `This will mark “Acme Corp - Time & Materials (T&M)” as inactive. Contracts cannot be edited or deleted after creation.`
- Confirm action calls `PATCH /contracts/:id/deactivate`
- On success: refresh row/list and show success toast
- Inactive rows should not show an active Deactivate button

### 4f. Read-only detail view

Because the API includes `GET /contracts/:id`, FE should provide a read-only detail surface (drawer, modal, or dedicated page).

Recommended behavior:

- clicking the contract name or row opens a read-only detail drawer
- the detail surface displays:
  - computed name
  - customer
  - type label
  - description
  - status
  - created at
  - expires at
  - full type-specific financial fields
- no edit button and no delete button
- if contract is active, a Deactivate action may be repeated from the detail surface

### 4g. Empty, loading, and error states

- Empty active state: show “No active contracts yet” with Create Contract CTA
- Empty filtered state: show “No contracts match the current filters”
- Loading state: skeleton rows/spinner consistent with existing tables
- API errors: show inline toast/banner; preserve filters and modal form state when possible

## 5) Acceptance Criteria

### Data / API

- [ ] A contract can be created for any existing customer using any of the 4 supported contract types.
- [ ] New contracts are created with `status = active` and a system-generated `created_at`.
- [ ] The API exposes only: `GET /contracts`, `GET /contracts/:id`, `POST /contracts`, and `PATCH /contracts/:id/deactivate`.
- [ ] The API does **not** expose `PUT /contracts/:id`, generic `PATCH /contracts/:id`, or `DELETE /contracts/:id`.
- [ ] Creating a second active contract with the same `(customer_id, type)` returns HTTP `409 Conflict` with a clear message.
- [ ] After deactivation of an active contract, a new active contract of the same type can be created for the same customer.
- [ ] Contract `name` is returned by the API as a computed field in the format `{customerName} - {contractTypeLabel}`.
- [ ] Type-specific fields are validated correctly for each contract type.
- [ ] Fields not applicable to the selected type are not required and are not used by the system.
- [ ] `currency` is required wherever monetary terms are present and is stored as a 3-character ISO code.

### Business behavior

- [ ] Once created, a contract cannot be edited through either UI or API.
- [ ] Once created, a contract cannot be deleted through either UI or API.
- [ ] A contract can be deactivated, and its status changes to `inactive`.
- [ ] `expires_at` is optional and does not automatically deactivate the contract.
- [ ] Active/inactive uniqueness is enforced both at DB level and API level.

### Frontend

- [ ] The sidebar contains a Contracts navigation item.
- [ ] The Contracts page shows active contracts by default.
- [ ] Switching the status toggle to **All** shows inactive contracts as well.
- [ ] The grid can be filtered by customer.
- [ ] The grid shows computed Name, human-readable Type, Description, Commercial Terms, Status, Created At, Expires At, and Actions.
- [ ] Type-specific financial fields are visible in the grid summary and in the read-only detail surface.
- [ ] The Create Contract button opens a modal consistent with the existing customer creation pattern.
- [ ] The contract creation flow adapts to the selected contract type and only shows the relevant commercial fields.
- [ ] Active rows expose a Deactivate action.
- [ ] No Edit or Delete actions are shown anywhere on the Contracts screen.
- [ ] On successful create or deactivate, the grid refreshes and reflects the latest contract state.
